# Master Documents

## 1. プロダクト概要

本プロジェクトは、海洋プラスチックごみの回収と海洋生物の保護を題材にした、マルチエージェント型の環境シミュレーションゲームである。  
展示用途を想定し、中高生や一般来場者が「ロボットの役割分担」と「環境負荷のトレードオフ」を視覚的に理解できることを重視する。

## 2. ターゲット

- 5月祭を訪れる中高生
- 技術・ロボティクスに関心のある一般層
- 環境問題や海洋保全に関心のある来場者

## 3. コア体験

ユーザーはロボット構成と行動パラメータを調整し、限られたステップ数とバッテリーの中で、以下のバランスを最適化する。

- 海洋ごみをできるだけ多く回収する
- 海洋生物へのストレスをできるだけ抑える
- ロボット同士の衝突や無駄移動を減らす
- 役割分担によってチーム全体の性能を高める

## 4. ゲームルール

### 4.1 ステージ

- ステージは 2D 平面またはセル分割された盤面として扱う
- 初期実装では連続座標ベースでもよいが、論理上は `W x H` の環境サイズを持つ
- 環境は以下の状態を保持する
  - ごみ配置
  - ロボット配置
  - 海洋生物配置
  - 各種統計値

### 4.2 エージェント

#### 海洋生物

- 初期配置は 10 体前後
- 基本移動はランダムウォークまたは群れ行動
- ロボットが回避半径内に入ると離れる
- 回避行動の回数や継続時間に応じてストレス値が増加する
- ストレス値が閾値を超えると消滅、または生態系スコアへ大きなペナルティを与える

#### スカウトロボット

- ごみを探索する
- 自身では回収しない
- 一定周期で周辺の占有状態を観測し、回収ロボットへ送信する
- バッテリーを持ち、移動やスキャンで消費する

#### 回収ロボット

- スカウトより狭い範囲の近接センサーを持つ
- ごみの回収ができる
- 共有されたターゲット情報をもとに移動する
- バッテリー切れ時は停止し、回復条件を満たすまで復帰できない

## 5. 行動モデル

### 5.1 ターン処理

1. 全ロボットがセンサー情報を取得する
2. スカウトが検知したごみ情報を共有する
3. 各ロボットが進行ベクトル `V_next` を計算する
4. 移動結果に応じて回収数、衝突数、バッテリー残量を更新する
5. 海洋生物の移動とストレス更新を行う
6. スコアを再計算する

### 5.2 進行ベクトル

ロボットの移動は重み付きベクトル合成で決定する。

`V_next = w_trash * V_trash + w_avoid_fish * V_avoid_fish + w_avoid_robot * V_avoid_robot + w_random * V_random`

- `V_trash`: ごみへ向かうベクトル
- `V_avoid_fish`: 海洋生物を避けるベクトル
- `V_avoid_robot`: 他ロボットとの衝突を避けるベクトル
- `V_random`: 探索や停滞回避のためのランダム成分

## 6. スコアリング

最終スコア `S` は、展示用途でも説明しやすい線形結合を基本とする。

`S = a * N_trash - b * N_collision - c * P_stress + d * E_remaining`

- `N_trash`: 回収したごみ数
- `N_collision`: ロボット衝突回数
- `P_stress`: 海洋生物ストレス総量
- `E_remaining`: 最終残バッテリー総量

## 7. 必要機能

- ステージサイズとステップ数の設定
- ロボット種別ごとの初期配置
- ごみの自然流入またはランダム配置
- 海洋生物の移動、回避、ストレス計測
- ロボットの観測、情報共有、移動、回収、バッテリー管理
- スコア計算
- リアルタイム描画
- 設定パラメータの UI 編集
- 実行中統計の可視化
- リセットと再実行

## 8. システム設計原則

`system_making_plan.md` の方針に基づき、以下を原則とする。

- ロボット種別が増えても API と内部 I/O はなるべく共通化する
- ロボット固有機能はコンポーネント追加で拡張できるようにする
- 個別ロボットの独立性と、チームとしての協調動作を両立する
- 後続メンバーがデザイン、挙動、モーション、パラメータを段階的に差し替えられる構造にする
