# First Implementation Plan

## 1. 目的

現在の海洋シミュレーション基盤を、展示向けの「ロボットによる海洋ごみ回収シミュレーション」へ拡張可能な構成に再定義する。  
この計画書は、プロジェクトメンバー全員が同じ前提で作業できるよう、実装順序、責務分担、成果物を明文化するものである。

## 2. 完成条件

以下を満たした時点で、最初の設計フェーズを完了とする。

- 展示向けゲームの仕様が `docs/master_documents.md` に統合されている
- 技術構成と拡張方針が設計文書化されている
- API とデータモデルの統一方針が定義されている
- 後続メンバーが各ロボットの見た目や挙動を独立に拡張できる
- 説明用スライドと PDF が存在する

## 3. 前提

- 現行リポジトリは FastAPI + React によるリアルタイムシミュレーション基盤を持つ
- 既存エージェントは `fish / predator / plastic` だが、今後は `scout / collector / marine_life / trash` へ役割整理する
- 直近では DB 未使用でもよいが、将来拡張に備えて永続化モデルを先に定義する

## 4. フェーズ

### Phase 1: 設計確定

- master document の再整理
- ロボット種別、状態、観測情報、共有情報の定義
- スコアとバッテリー仕様の明文化

### Phase 2: バックエンド構造整備

- エージェント共通 I/O モデルを整理
- ロボット種別ごとのロジック差分をコンポーネント化
- シミュレーション状態を API / WebSocket へ統一形式で出力

### Phase 3: フロントエンド整備

- ロボット種別ごとの描画差し替えを renderer registry で継続対応
- パラメータ UI を役割別に整理
- スコア、バッテリー、生物ストレスを常時表示

### Phase 4: チーム拡張対応

- 見た目変更用ファイルを分離
- ロボットモーション追加箇所を明確化
- 個別ロボット挙動のパラメータ差分だけで調整できる構造へ寄せる

## 5. 実装タスク一覧

### ドメイン設計

- ロボット役割の再定義
- 海洋生物ストレスモデルの確定
- ごみ検知と共有メッセージ仕様の確定
- スコア計算式の固定

### バックエンド

- 共通エージェントベースクラスの責務整理
- センサー結果と行動決定結果のデータ構造定義
- ロボット間共有メッセージの導入
- 将来 DB 保存用のイベントモデル定義

### フロントエンド

- 状態パネルにゲーム目的を反映
- 役割別の色分けと凡例の追加
- スコア変動要因の可視化
- 設定入力のラベル整理

### ドキュメント

- 実装計画書
- 技術構成書
- API / DB 設計書
- スライド資料

## 6. チーム分担の考え方

- PM 相当の 2 名は全体仕様、構造、レビュー責務を持つ
- 後続の 3 名は、共通 I/O を壊さない前提で、デザイン、モーション、挙動調整、追加機能を担当する
- そのため、個別実装者が変更してよい範囲を `renderer`、`agent component`、`parameter config` に分離する

## 7. リスク

- 現状コードは「生態系シミュレーション」の責務が強く、ゲーム仕様と名称が一致していない
- ロボット間通信を雑に実装すると、後続メンバーが機能追加しづらくなる
- DB を後付けにすると履歴再生や分析機能の設計変更コストが大きくなる

## 8. 優先順位

1. 共通データ構造の定義
2. ロボット役割の分離
3. UI とスコア仕様の見える化
4. 永続化用の設計
5. 演出やモーション強化

## 9. 当面の実装指針

- バックエンドは「環境」「エージェント」「共有メッセージ」「スコア」を中心に整理する
- フロントエンドは「描画」「操作」「統計」の 3 層を維持する
- 追加ロボットは既存 registry / base class へ接続する
- すべての追加機能は、後続メンバーが単独で差し替えられる単位に分割する
